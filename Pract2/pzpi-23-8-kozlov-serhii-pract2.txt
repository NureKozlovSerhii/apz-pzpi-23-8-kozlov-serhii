3 ОПИС ВИКОНАНОЇ РОБОТИ




3.1 Загальні відомості про систему 


Назва системи: Spotify.
Розробник: Spotify AB. 
Призначення системи: Spotify — це глобальна аудіострімінгова платформа, що надає доступ до мільйонів музичних треків та подкастів. Цільова аудиторія охоплює понад 500 мільйонів активних користувачів щомісяця у всьому світі. Головним інженерним викликом для платформи є необхідність забезпечення безперервного потокового відтворення аудіо з мінімальною затримкою (latency), незалежно від географічного розташування користувача та якості його інтернет-з'єднання. Система повинна обробляти мільйони запитів на секунду (пошук, відтворення, лайки), зберігаючи при цьому високу доступність та консистентність даних.


3.2 Архітектурний стиль системи 


Для забезпечення безпрецедентної масштабованості та надійності, Spotify використовує гібридний архітектурний стиль. Основою системи є мікросервісна архітектура (Microservices Architecture). Вся платформа розділена на понад 1500 незалежних мікросервісів. Обґрунтування вибору: монолітна архітектура не дозволила б розробникам швидко впроваджувати нові функції та масштабувати окремі вузли (наприклад, лише сервіс пошуку під час пікових навантажень). Мікросервіси забезпечують ідеальну ізоляцію збоїв: якщо сервіс рекомендацій виходить з ладу, основний функціонал (відтворення збережених треків) продовжує працювати без перебоїв. Додатково активно застосовується подієво-орієнтована архітектура (Event-Driven Architecture) на базі Apache Kafka для забезпечення асинхронної взаємодії між сервісами


3.3 Основні компоненти системи 


Архітектуру платформи можна умовно поділити на декілька фундаментальних рівнів:
* Інтерфейс користувача (Клієнти): Мобільні, десктопні та веб-додатки. Клієнти Spotify відрізняються "агресивним" механізмом локального кешування. Вони завантажують наступні треки у фоновому режимі ще до того, як користувач натисне кнопку "Next", що створює ілюзію миттєвого відтворення.
* API Gateway (Серверна частина): Єдина точка входу (Single Point of Entry) для всіх запитів. Виконує роль фасаду (BFF - Backend for Frontend), забезпечуючи маршрутизацію запитів, аутентифікацію (OAuth), обмеження швидкості (Rate Limiting) та первинну перевірку безпеки.
* Рівень бізнес-логіки: Набір незалежних мікросервісів, кожен з яких відповідає за свою предметну область (Domain). Наприклад: сервіс управління плейлистами, сервіс пошуку, сервіс білінгу, сервіс формування рекомендацій (Discover Weekly).
* Рівень зберігання даних: У Spotify реалізовано підхід Polyglot Persistence — використання різних типів баз даних для різних потреб. Для збереження масивів даних користувачів (історія прослуховувань) використовується розподілена NoSQL СУБД Cassandra. Для транзакційних даних (інформація про оплату) — PostgreSQL.
* Допоміжні сервіси: Глобальна мережа доставки контенту (CDN) для роздачі самих аудіофайлів та кластери Memcached/Redis для миттєвого доступу до метаданих пісень.




3.4 Взаємодія компонентів 


Процес взаємодії компонентів суворо регламентований і поділяється на зовнішній та внутрішній трафік. Клієнтські додатки спілкуються з API Gateway переважно за допомогою протоколів HTTP/HTTPS через RESTful API та GraphQL. GraphQL дозволяє мобільному клієнту отримати всю необхідну інформацію для побудови екрану (ім'я артиста, обкладинка альбому, список треків) за один мережевий запит.


Після того, як запит потрапляє всередину захищеного периметра системи, характер взаємодії змінюється. Мікросервіси спілкуються між собою за допомогою протоколу gRPC. Він працює поверх HTTP/2 та використовує бінарний формат серіалізації Protocol Buffers (Protobuf). Це дозволяє зменшити розмір пакетів та в рази підвищити швидкість обміну даними між внутрішніми вузлами системи порівняно з класичним JSON.


Окремим шаром взаємодії є асинхронні потоки даних (Data Pipelines). Коли користувач натискає "Play", клієнт не тільки запитує аудіофайл, але й генерує подію. Ця подія відправляється у шину повідомлень Apache Kafka. Десятки інших мікросервісів (сервіс виплат артистам, аналітика, рекомендаційна система) є підписниками (Consumers) відповідних топіків у Kafka. Вони зчитують події у своєму власному темпі, не навантажуючи основний потік виконання відтворення музики.


  

Опис діаграми: На діаграмі відображено потік даних від мобільного клієнта до API Gateway, подальший синхронний виклик мікросервісу відтворення через gRPC та асинхронну публікацію події в кластер Apache Kafka для подальшої обробки сервісом аналітик




3.5 Масштабованість та надійність 


Архітектура Spotify розроблена з урахуванням горизонтального масштабування.
* Контейнеризація та оркестрація: Усі мікросервіси упаковані у Docker-контейнери. Для управління цим парком контейнерів використовується Kubernetes (K8s). У разі різкого збільшення трафіку (наприклад, під час релізу нового альбому відомого виконавця), Kubernetes автоматично розгортає додаткові екземпляри (Pods) навантаженого мікросервісу.
* Балансування навантаження: Трафік розподіляється між сотнями серверів за допомогою інтелектуальних балансувальників навантаження (Load Balancers), які враховують поточний стан та "здоров'я" (Health Checks) кожного вузла.
* Кешування та CDN: Самі музичні файли не генеруються і не роздаються бекенд-серверами компанії. Вони розміщені на сторонніх і власних серверах Content Delivery Network (CDN) по всьому світу. Коли користувач запитує пісню, вона завантажується з фізично найближчого до нього сервера, що радикально знижує латентність та знімає навантаження з дата-центрів компанії.




3.6 Приклади програмної реалізації 


Для ілюстрації архітектурних принципів (зокрема, мікросервісного підходу та подієво-орієнтованої взаємодії) було розроблено програмний код мовою Python (див. Додаток В). Код включає створення API Gateway з використанням фреймворку FastAPI, реалізацію продюсера (Producer) для публікації подій у шину Apache Kafka, а також консюмера (Consumer), який імітує мікросервіс аналітики та обробляє дані з топіку.

Файл 1: kafka_producer.py (Модуль публікації подій)


1  import json
2  from kafka import KafkaProducer
3  
4  class SpotifyEventProducer:
5    def __init__(self, broker_url='localhost:9092'):
6      self.topic = 'spotify-user-activity'
7      try:
8        self.producer = KafkaProducer(
9          bootstrap_servers=[broker_url],
10         value_serializer=lambda v: json.dumps(v).encode('utf-8')
11       )
12       self.is_connected = True
13     except Exception as e:
14       print(f"[Producer] Неможливо підключитися до Kafka. Режим імітації.")
15       self.is_connected = False
16 
17   def emit_event(self, event_data: dict):
18     if self.is_connected:
19       self.producer.send(self.topic, event_data)
20       self.producer.flush()
21     print(f"[Event Producer] Подія '{event_data['event_type']}' надіслана у шину.")

1  import uuid
2  from datetime import datetime
3  from fastapi import FastAPI, HTTPException, BackgroundTasks
4  from pydantic import BaseModel
5  from kafka_producer import SpotifyEventProducer
6  
7  app = FastAPI(title="Spotify API Gateway Mock", version="1.0")
8  event_bus = SpotifyEventProducer()
9  
10 class TrackRequest(BaseModel):
11   user_id: str
12   track_id: str
13 
14 @app.post("/api/v1/stream/play")
15 async def play_audio_stream(request: TrackRequest, background_tasks: BackgroundTasks):
16   if not request.user_id or not request.track_id:
17     raise HTTPException(status_code=400, detail="Відсутні параметри")
18 
19   event_payload = {
20     "event_id": str(uuid.uuid4()),
21     "event_type": "TRACK_PLAYED",
22     "user_id": request.user_id,
23     "track_id": request.track_id,
24     "timestamp": datetime.utcnow().isoformat()
25   }
26 
27   # Фонова відправка події, щоб не блокувати відповідь клієнту
28   background_tasks.add_task(event_bus.emit_event, event_payload)
29 
30   return {
31     "status": "STREAM_STARTED",
32     "cdn_url": f"https://cdn.spotify.mock/audio/{request.track_id}.mp3",
33     "latency_ms": 42
34   }

2  from kafka import KafkaConsumer
3  
4  class CassandraMockDB:
5    def __init__(self):
6      self.play_history = []
7    def insert_event(self, event: dict):
8      self.play_history.append(event)
9      print(f"[Cassandra DB] Запис збережено. Всього: {len(self.play_history)}")
10 
11 class AnalyticsWorker:
12   def __init__(self, broker_url='localhost:9092'):
13     self.topic = 'spotify-user-activity'
14     self.db = CassandraMockDB()
15     try:
16       self.consumer = KafkaConsumer(
17         self.topic,
18         bootstrap_servers=[broker_url],
19         value_deserializer=lambda m: json.loads(m.decode('utf-8')),
20         group_id='analytics-consumer-group'
21       )
22     except Exception:
23       self.consumer = None
24 
25   def run(self):

27     if not self.consumer:
28       return
29     for message in self.consumer:
30       event = message.value
31       if event.get('event_type') == 'TRACK_PLAYED':
32         print(f"[Analytics] Оброблено подію від користувача {event['user_id']}")
33         self.db.insert_event(event)
34         
35 if __name__ == "__main__":
36   worker = AnalyticsWorker()
37   worker.run()
