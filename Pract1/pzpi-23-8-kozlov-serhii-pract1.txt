3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування 
Шаблони (патерни) проєктування — це перевірені часом, стандартизовані архітектурні підходи до розв'язання поширених проблем проєктування об'єктно-орієнтованого програмного забезпечення. Вони не є готовими фрагментами коду, а слугують загальною концепцією вирішення певного класу задач. Базовим стандартом у цій галузі є класифікація «Банди чотирьох» (GoF), згідно з якою 23 класичні патерни поділяються на три групи: твірні (оптимізують створення об'єктів), структурні (визначають способи побудови складних систем) та поведінкові (регламентують алгоритми взаємодії). Головна область застосування патернів полягає у створенні гнучкої, масштабованої архітектури, яка легко піддається супроводу.


3.2 Опис обраного патерна
   * Назва патерна: Спостерігач (Observer)
   * Тип шаблона (патерна): поведінковий
   * Основне призначення: створення механізму підписки, який дозволяє одному об'єкту (видавцю) автоматично повідомляти безліч інших зацікавлених об'єктів (підписників) про будь-які зміни свого внутрішнього стану.
   * Передумови застосування: патерн доцільно використовувати, коли зміна стану одного компонента вимагає оновлення інших, причому кількість та конкретні класи цих компонентів можуть бути невідомими заздалегідь. Типові сценарії: розробка UI-віджетів, реалізація архітектури MVC, а також побудова подієво-орієнтованих систем (Event-Driven Architecture).
   * Переваги: суворе дотримання принципу Open/Closed Principle (OCP), оскільки можна додавати нових підписників без зміни коду видавця; забезпечення слабкої зв'язності (loose coupling) між об'єктами; можливість динамічної реєстрації та відписки під час виконання програми.
   * Недоліки: ризик виникнення витоків пам'яті (проблема «lapsed listener» / забутого слухача), якщо об'єкти не відписуються від подій; випадковий порядок сповіщень, що може ускладнити відстеження логіки; складність налагодження (дебагінгу) у разі виникнення ланцюгових (каскадних) оновлень.


3.3 Структура шаблона (патерна)
   1. Опис структурних частин:
   * Видавець (Subject): інтерфейс або абстрактний клас, що визначає методи для приєднання (attach), від'єднання (detach) та сповіщення (notify) підписників.
   * Конкретний Видавець (ConcreteSubject): клас, що зберігає стан, який становить інтерес для підписників. При зміні цього стану він викликає метод notify.
   * Підписник (Observer): загальний інтерфейс для всіх об'єктів, які мають реагувати на оновлення. Зазвичай містить єдиний метод update.
   * Конкретний Підписник (ConcreteObserver): реалізує інтерфейс Observer, зберігає посилання на ConcreteSubject (за потреби) та містить конкретну логіку реагування на подію.
   2. Опис взаємодії об’єктів: Клієнтський код змінює стан об'єкта ConcreteSubject. Останній самостійно ітерує внутрішній список своїх підписників та викликає метод update у кожного об'єкта типу Observer, передаючи йому контекст події або оновлені дані.


  

На діаграмі зображено зв'язок агрегації між Subject та Observer (видавець містить список підписників), а також зв'язок реалізації між інтерфейсами та конкретними класами.


3.4 Реалізація шаблона (патерна)
   1. Опис реалізації мовою програмування Python
   2. Основні етапи:
   * Створення абстрактних класів: використання модуля abc для оголошення абстрактного базового класу Observer з методом update.
   * Реалізація Видавця: створення класу Publisher із внутрішнім списком _observers та методами attach, detach, notify.
   * Реалізація конкретних підписників: розробка класів EmailSubscriber та PushSubscriber, кожен з яких має власну імплементацію реакції (наприклад, виведення логів про відправку email або push-сповіщення).
   * Демонстрація використання: у клієнтському коді створюється екземпляр Publisher, до якого динамічно підписуються кілька слухачів. Виклик методу зміни стану призводить до автоматичного сповіщення всіх зареєстрованих об'єктів.

1  # 1. Абстрактні базові класи та інтерфейси
2  from abc import ABC, abstractmethod
3  
4  # Інтерфейс підписника (Observer)
5  class Observer(ABC):
6      @abstractmethod
7      def update(self, message: str) -> None:
8          pass
9  
10 # 2. Видавець (Subject / Publisher)
11 class Publisher:
12     def __init__(self):
13         # Список для зберігання підписників
14         self._observers: list[Observer] = []
15 
16     def attach(self, observer: Observer) -> None:
17         self._observers.append(observer)
18 
19     def detach(self, observer: Observer) -> None:
20         self._observers.remove(observer)
21 
22     def notify(self, message: str) -> None:
23         print("\n[Publisher] Сповіщаю всіх підписників про нову подію...")
24         for observer in self._observers:
25             observer.update(message)
26 
27     def do_business_logic(self) -> None:
28         print("\n[Publisher] Виконую важливу бізнес-логіку: Створено нового користувача.")
29         # Зміна стану викликає сповіщення
30         self.notify("Користувача 'JohnDoe' успішно зареєстровано.")
31 
32 # 3. Конкретні підписники (Concrete Observers)
33 class EmailSubscriber(Observer):
34     def update(self, message: str) -> None:
35         print(f"EmailSubscriber: Відправлено email з текстом: {message}")
36 
37 class PushSubscriber(Observer):
38     def update(self, message: str) -> None:
39         print(f"PushSubscriber: Відображено push-сповіщення: {message}")
40 
41 # 4. Клієнтський код (Client)
42 if __name__ == "__main__":
43     # Ініціалізація видавця
44     publisher = Publisher()
45 
46     # Створення підписників
47     email_sub = EmailSubscriber()
48     push_sub = PushSubscriber()
49 
50     # Підписка на події
51     publisher.attach(email_sub)
52     publisher.attach(push_sub)
53 
54     # Виконання дії, яка автоматично запустить розсилку сповіщень
55     publisher.do_business_logic()
